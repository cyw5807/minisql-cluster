## 4.1 SQL 功能测试

### 【SQL-FUNC-01】创建新表并持久化

#### 📋 前置环境

- 启动本地 ZooKeeper
- 确保 Master 进程正常运行并连接至 ZK

#### 🔧 实操步骤

1. 在代码或客户端中，手动构造一个包含字段（`ColumnMeta`）及主键的 `TableMeta` 对象
2. 主动调用 `MetadataManager.createTable()` 方法触发建表流程

#### ✅ 验收标准

- [ ] 使用 ZK 客户端（如 PrettyZoo 或 zkCli）查看，确认成功生成了 `/minisql/metadata/表名` 节点
- [ ] 确认该节点存储的序列化数据能够被反序列化，精准还原表结构
- [ ] 全程不抛出异常

---

### 【SQL-FUNC-02】元数据重名表创建拦截

#### 📋 前置环境

- 确保目标表名已经存在于集群的元数据中

#### 🔧 实操步骤

1. 再次发送与已存在表同名的 `CREATE TABLE` 请求

#### ✅ 验收标准

- [ ] 系统必须拒绝执行
- [ ] 控制台捕获到 `RuntimeException`，并明确提示"表已存在"
- [ ] 检查 ZK，确认原有表结构数据未被覆盖篡改

---

### 【SQL-FUNC-03】全链路 SQL 解析与下发

#### 📋 前置环境

- 1 个 Master 和 2 个 Worker 组成的集群均已就绪

#### 🔧 实操步骤

1. 在客户端发送查询语句：
   ```sql
   SELECT * FROM TEST_TABLE;
   ```
2. 观察整个集群的控制台日志流转

#### ✅ 验收标准

- [ ] 确认 SQL 被成功封装为 `RpcRequest`
- [ ] 确认客户端直接与 Worker 建立了连接并发送了任务
- [ ] 对应的 Worker 控制台成功打印出"接收并解析 SQL"

## 4.2 集群管理测试

### 【CLUSTER-01】动态扩容感知

#### 📋 前置环境

- Master 节点的雷达已启动
- 当前没有任何 Worker，可用列表为空

#### 🔧 实操步骤

1. 启动第一个 Worker 进程（端口配置为 **9011**）
2. 修改配置文件的端口，启动第二个 Worker 进程（端口配置为 **9012**）

#### ✅ 验收标准

- [ ] Master 的控制台应实时打印出两次【动态扩容】的日志
- [ ] Master 内存中的可用列表准确更新为包含这 2 个 `IP:Port` 的集合

---

### 【CLUSTER-02】元数据根目录自动初始化

#### 📋 前置环境

- 清空或启动一个全新的 ZK 集群
- 确保内部没有任何相关路径

#### 🔧 实操步骤

1. 启动 Master 节点进程

#### ✅ 验收标准

- [ ] Master 启动时不抛出 `NoNodeException`
- [ ] 自动在 ZK 中创建好两个持久化父节点：
  - `/minisql/workers`
  - `/minisql/metadata`

## 4.3 容错容灾测试

### 【HA-01】Master 脑裂防范与主备切换

#### 📋 前置环境

- 集群中已经存在 1 个处于"活跃"状态的 Master

#### 🔧 实操步骤

1. 启动第二个 Master 进程
2. 强制杀掉（Kill）第一个活跃 Master 进程

#### ✅ 验收标准

- [ ] 第二个 Master 刚启动时，控制台需提示"已存在活跃 Master"并自动降级为备用状态
- [ ] 主节点被杀掉约 **30 秒**后，备用节点瞬间接管，并打印出"选举成功"的日志

---

### 【HA-02】Worker 宕机时的剔除与路由保护

#### 📋 前置环境

- 集群中存在两台活跃的 Worker

#### 🔧 实操步骤

1. 通过强制 Kill 的方式，终止其中一台 Worker 进程
2. 立即从 Master 发送连续的 SQL 任务进行抗压测试

#### ✅ 验收标准

- [ ] Master 必须在 **5 秒内**打印出【故障感知】Worker 下线的日志
- [ ] 后续发送的所有 SQL 请求必须 **100%** 路由至剩下的那台健康 Worker 上
- [ ] 期间无任何任务失败

---

## 4.4 负载均衡测试

### 【ROUTE-01】轮询调度算法准确性测试

#### 📋 前置环境

- 启动 1 个 Master 和 2 个 Worker

#### 🔧 实操步骤

1. 在客户端侧，实例化一个带有可用节点列表缓存的 `TaskRouter`
2. 连续发起 **6 次**独立的 RPC 模拟请求

#### ✅ 验收标准

- [ ] 客户端需直接向后端发起连接
- [ ] 通过控制台日志确认：6 个任务被两个 Worker 交替认领执行，而不是某个 Worker 接收所有任务
- [ ] 验证执行期间，Master 节点无任何网络流量介入

---

### 【ROUTE-02】无可用节点的熔断处理

#### 📋 前置环境

- 确保所有 Worker 节点均处于下线状态

#### 🔧 实操步骤

1. 尝试通过客户端发送任意查询请求

#### ✅ 验收标准

- [ ] 路由层必须立刻拦截请求，并抛出致命异常：
  ```
  集群中没有任何存活的 Worker
  ```
- [ ] 底层的网络通信模块不应触发无意义的超时挂起等待

---

## 4.5 客户端测试

### 【CLIENT-01】TCP 端口防冲撞检测

#### 📋 前置环境

- 第一台 Worker 已经启动并占用了本地的 **9011** 端口

#### 🔧 实操步骤

1. 故意不修改端口配置，强行启动第二个 Worker 实例

#### ✅ 验收标准

- [ ] 底层 Netty 必须成功捕获到 `java.net.BindException`
- [ ] 控制台提示"端口被占用"并退出程序
- [ ] **绝对不允许**产生后台僵尸进程

---

### 【CLIENT-02】Worker 远端异常回传

#### 📋 前置环境

- 全链路网络连接正常

#### 🔧 实操步骤

1. 客户端故意发送一条必定会引发运行时错误的 SQL，例如：
   - 除以零
   - 违反主键冲突的插入语句

#### ✅ 验收标准

- [ ] 接收该语句的 Worker 进程**不能崩溃**
- [ ] 该异常被完美捕获，封装在 `RpcResponse.error` 字段中回传
- [ ] 最终在客户端侧抛出显示
