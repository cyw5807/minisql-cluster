# 🛡️ 基础测试一：控制中枢的"生死切换"测试

## 🎯 测试目标

证明当主控节点（Master）突然断电死机时，备用节点能在 **30 秒内**自动接管，系统不会崩溃。

---

## 🟢 准备工作（你只需要打开两个软件）

1. 打开你用来敲黑底白字命令的终端（CMD 或者 PowerShell）。
2. 打开你的写代码的软件（VS Code 或者 IDEA）。

---

## 🟢 第一步：启动系统的"心跳"（ZooKeeper）

### 操作说明

在你的终端里，进入你电脑上安装 ZooKeeper 的那个文件夹，运行启动命令。

- **Windows 用户**：双击 `bin` 目录下的 `zkServer.cmd`。
- **Mac 用户**：在终端输入以下命令：

```bash
./bin/zkServer.sh start
```

### ✅ 成功标志

- **Windows**：会弹出一个黑框框里面有一堆白色英文不停滚动。
- **Mac**：会提示 `STARTED`。

> ⚠️ **注意**：这个黑框框千万不要关！把它最小化藏起来就好。

---

## 🟢 第二步：唤醒"一号大管家"（启动主 Master）

### 操作步骤

1. 在你的代码软件左侧的文件树里，找到并双击打开以下文件：

   ```
   minisql-master/src/test/java/com/zju/minisql/master/zk/MasterElectionSimulator.java
   ```

2. 找到代码里的 `public static void main(String[] args)` 这一行。
3. 点击这行代码旁边的小小的绿色播放三角形（Run 按钮）。

### ✅ 成功标志

你会在屏幕下方的控制台看到一行字：

```
=== 选举成功！当前节点已成为活跃 Master... ===
```

---

## 🟢 第三步：唤醒"二号替补管家"（启动备用 Master）

### 操作步骤

1. **不要关掉**刚才运行的"一号大管家"！
2. 就在刚才那个 `MasterElectionSimulator.java` 文件的 `main` 方法旁边，再次点击那个绿色的播放三角形（Run 按钮）。
   > （这就相当于我们在同一台电脑上，又开了一个全新的大管家进程。）

### ✅ 成功标志

在新弹出的控制台里，你会看到这样的提示：

```
=== 选举失败。当前集群已有活跃 Master... ===
本节点降级为备用 Master，正在监听 Leader 状态...
```

### 📊 目前状态

现在，你的电脑里同时住着一主一备两个管家，备用管家正在默默盯着主干管家。

---

## 🟢 第四步：拔掉"一号大管家"的网线（模拟死机宕机）

### 操作步骤

1. 在你的代码软件的控制台面板上，切换回到"一号大管家"的那个输出窗口。
2. 找到控制台旁边的一个红色的正方形按钮（Stop / Terminate 按钮），毫不犹豫地点下去！

### ✅ 成功标志

"一号大管家"的控制台提示：

```
Process finished with exit code...
```

彻底死透了。

---

## 🟢 第五步：见证奇迹（验证自动接管）

### 操作步骤

1. 马上切换到"二号替补管家"的控制台窗口。
2. **什么都不要按**，在心里默默数 **30 个数**（这是系统设置的确认死机的缓冲时间）。

### ✅ 成功标志

30 秒后，原本安静的"二号管家"控制台会突然弹出新消息：

```
!!! 警告: 检测到活跃 Master 宕机...
=== 选举成功！本节点晋升为活跃 Master ===
```

---

# 🛡️ 基础测试二：手脚就绪（Worker 注册测试）

## 🟢 准备工作

请看一眼你电脑下面的任务栏，确认一下第一关里打开的 ZooKeeper（那个黑色的终端窗口）还在运行。只要它没关，准备工作就满分通过。

---

## 🟢 第一步：找到"一号打工人"的启动文件

在你写代码的软件（VS Code 或 IDEA）左侧的文件树里：

1. 点开 `minisql-worker` 文件夹。
2. 顺着目录往下找，找到并双击打开这个文件：

   ```
   src/main/java/com/zju/minisql/worker/WorkerStarter.java
   ```

---

## 🟢 第二步：唤醒"一号打工人"

在打开的 `WorkerStarter.java` 文件里：

1. 找到 `public static void main(String[] args)` 这一行。
2. 点击这行代码旁边的小小的绿色播放三角形（Run 按钮）。

---

## 🟢 第三步：检查"打工人"的体检报告（验证结果）

稍微等两三秒钟，把目光移到屏幕下方的控制台，如果你看到了类似下面这几行字，就说明测试成功了！

你会依次看到：

```
本地服务初始化完成。
已加载演示数据，并注册分布式查询执行服务。
```

接着会有一些底层框架的运行日志（不用管细节），最后它会停住，程序不会退出。这就说明它的内部通信基站（Netty 端口 `9012`）已经稳稳地建立起来了，正在乖乖地竖起耳朵，等待接收任务。

---

## 💡 进阶小魔术：如何确认它真的去 ZK 报到了？

你可能会问："控制台只说它启动了，我怎么知道它有没有骗我，它真的去 ZooKeeper 登记了吗？"

还记得我们在上一步那个因为找不到 Worker 而报错的 `GroupBQueryDemo.java` 吗？现在就是用它来验证的最好时机！

---

## 🟢 第四步：用"大脑"去呼叫"手脚"

> ⚠️ **重要**：千万不要关掉刚才启动的 `WorkerStarter`！让它在后台开着。

在左侧文件树里，找到我们之前写的那个"大脑"：

```
minisql-client/src/main/java/com/zju/minisql/client/demo/GroupBQueryDemo.java
```

点击它的 `main` 方法旁边的绿色播放三角形（Run 按钮）。

---

## 🟢 第五步：见证大团圆！

这一次，控制台绝对不会再报 `未发现任何 Worker，请先启动 WorkerStarter` 这个错误了！

相反，你应该会看到控制台打印出：

```
初始化可用 Worker 列表: [127.0.0.1:9012]
```

---

# 🚀 终极测试三：全链路大联调

## 🟢 战前状态确认（非常重要！）

在开始之前，请确认你的电脑目前处于以下状态（也就是前面两关的战果）：

- ✅ **ZooKeeper（黑框框）** 正在后台默默运行。
- ✅ **"一号打工人"（WorkerStarter）** 正在你的代码软件里运行，控制台安安静静地停在那里，没有报错退出。

---

## 🟢 第一步：找到"司令部"（Client 测试入口）

在你的代码软件左侧文件树中，找到并双击打开客户端的演示文件：

```
minisql-client/src/main/java/com/zju/minisql/client/demo/GroupBQueryDemo.java
```

> （这就是我们在测试二里，因为找不到 Worker 而报错的那个文件。现在 Worker 已经就绪，它要发挥真正的实力了！）

---

## 🟢 第二步：按下"发射按钮"

找到 `GroupBQueryDemo.java` 里的 `public static void main(String[] args)` 这一行。

点击这行代码旁边的小小的绿色播放三角形（Run 按钮）。

---

## 🟢 第三步：见证"大脑"的思考与分发（观察 Client 控制台）

程序运行后，请紧盯当前 Client 的控制台，你会像看电影一样看到以下壮观的流程：

1. **连接成功**：`初始化可用 Worker 列表: [127.0.0.1:9012]`（确认找到了打工人）。
2. **解析 SQL**：打印出那句复杂的 `SELECT ... JOIN ... GROUP BY` 语句。
3. **生成计划**：打印出 AST（抽象语法树）或者逻辑执行计划（Logical Plan）。
4. **全军出击（最关键的一步）**：你会看到类似以下日志：

   ```
   [RPC Client] 正在向 127.0.0.1:9012 发送分布式子任务 (Fragment)...
   ```

5. **等待结果**：客户端可能会提示 `正在等待 Worker 返回计算结果...`。

---

## 🟢 第四步：观察"手脚"的疯狂运转（切换到 Worker 控制台）

> ⏱️ **不要眨眼！** 立刻在你的代码软件的控制台面板上，切换到刚才一直在后台静静等待的"打工人"（WorkerStarter）的输出窗口。

如果你看到它原本安静的控制台突然弹出了新日志，比如：

```
[RPC Server] 接收到来自 Client 的执行请求: 分布式查询任务...
[执行引擎] 开始执行过滤 / 聚合 / JOIN 逻辑...
[RPC Server] 任务执行完毕，向 Client 返回局部结果 (PartialQueryResult)...
```

这就说明你的 **RPC 网络通信 100% 畅通无阻**，并且底层的 **Kryo 序列化完美工作**！

---

## 🟢 第五步：大丰收（切回 Client 控制台）

再次切回 Client（司令部）的控制台，如果一切顺利，你会看到它接收到了 Worker 发回来的数据，并在最后打印出类似这样的完美落幕：

```
[Client] 成功接收结果！
==================== 查询结果 ====================
| dept       | cnt |
| 计算机学院 |  15 |
| 法学院     |  12 |
=================================================
```
