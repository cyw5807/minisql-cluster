# A 组测试场景与用例说明

本文档针对 A 组四个模块的测试，逐条说明：

- 测试的 `Scenario`（场景）
- 测试的 `Use Case`（业务用例）
- 测试输入
- 预期输出 / 断言
- 对应脚本与测试类

---

## 1. 数据分布模块

### 1.1 基础路由 + 集群变更后可继续路由

- **Scenario**
  - 集群初始有 2 个可用节点。
  - 对同一个 key 做写路由。
  - 移除一个节点并触发 `onClusterChange`。
  - 再次对同一个 key 做写路由。
- **Use Case**
  - 验证一致性哈希路由在集群变更后不会失效，缓存会被正确失效并重建。
- **输入**
  - 初始节点：`127.0.0.1:9012`、`127.0.0.1:9013`
  - key：`user_1`
  - 事件：`ClusterEvent.removed(removedNode)`
- **预期输出**
  - 第一次 `routeForWrite("user_1")` 返回非空节点；
  - 节点变更后第二次 `routeForWrite("user_1")` 依旧返回非空节点；
  - 不抛异常。
- **对应实现**
  - 测试类：`minisql-common/src/test/java/com/zju/minisql/common/distribution/DistributionManagerImplTest.java`
  - 脚本：`scripts/test-a-distribution.ps1`

## 2. 副本管理模块

### 2.1 写入成功（副本确认通过）

- **Scenario**
  - 一个分片包含 1 个主节点 + 2 个副本节点。
  - 副本同步传输被 mock 为总是成功。
- **Use Case**
  - 验证写入路径能满足确认条件并返回成功结果。
- **输入**
  - partition：`1`
  - primary：`127.0.0.1:9012`
  - replicas：`127.0.0.1:9013`、`127.0.0.1:9014`
  - row：`Row.of("id", 1001, "name", "alice")`
- **预期输出**
  - `replicaManager.write(...)` 返回 `isSuccess == true`。

### 2.2 读取路由到可用节点

- **Scenario**
  - 使用自定义负载均衡器 `SimpleLoadBalancer` 选择可读节点。
- **Use Case**
  - 验证副本读请求可以经过副本路由组件返回有效目标节点。
- **输入**
  - partition：`1`
  - key：`"1001"`
- **预期输出**
  - `replicaManager.read(...)` 返回 `isSuccess == true`；
  - `message` 中包含节点地址（含 `127.0.0.1`）。

### 2.3 主节点故障后的主备切换

- **Scenario**
  - 当前主节点 `127.0.0.1:9012` 发生故障。
- **Use Case**
  - 验证故障回调后会选举新主并更新元数据。
- **输入**
  - 故障节点：`127.0.0.1:9012`
  - 调用：`replicaManager.onNodeFailure(...)`
- **预期输出**
  - `metadata.getPrimaryNode(partitionId)` 非空；
  - 新主节点 `nodeId != 127.0.0.1:9012`。
- **对应实现**
  - 测试类：`minisql-common/src/test/java/com/zju/minisql/common/replica/ReplicaManagerImplTest.java`
  - 脚本：`scripts/test-a-replica.ps1`

### 2.4 Primary 写入失败后自动切换并重试

- **Scenario**
  - 当前 primary 节点写入失败（模拟节点不可写），副本节点可写。
- **Use Case**
  - 验证 `ReplicaManager.write(...)` 在 primary 写失败时，能自动触发故障切换并在新主上重试写入。
- **输入**
  - partition：`2`
  - primary：`127.0.0.1:9012`（模拟写失败）
  - replicas：`127.0.0.1:9013`、`127.0.0.1:9014`（模拟可写）
  - row：`Row.of("id", 2001, "name", "bob")`
- **预期输出**
  - `replicaManager.write(...)` 返回 `isSuccess == true`；
  - `metadata.getPrimaryNode(partitionId).nodeId != 127.0.0.1:9012`。
- **对应实现**
  - 测试类：`minisql-common/src/test/java/com/zju/minisql/common/replica/ReplicaManagerImplTest.java`
  - 覆盖方法：`shouldAutoFailoverAndRetryWhenPrimaryWriteFails`

### 2.5 副本同步 RPC 传输打包正确

- **Scenario**
  - 使用假的 `FragmentTaskClient` 捕获传输内容，执行一次副本同步写入。
- **Use Case**
  - 验证 `ReplicaSyncRpcTransport` 会将写入转换为 `INSERT` 片段，并发送到目标副本地址。
- **输入**
  - 目标副本：`127.0.0.1:9013`
  - table：`student`
  - row：`Row.of("id", 1001, "name", "alice")`
- **预期输出**
  - `syncWrite(...) == true`；
  - 捕获到的 `TaskFragment.queryAst.statementType == "INSERT"`；
  - 捕获到的 `queryAst.tableName == "student"`。
- **对应实现**
  - 测试类：`minisql-client/src/test/java/com/zju/minisql/client/network/ReplicaSyncRpcTransportTest.java`
  - 脚本：`scripts/test-replica-fix.ps1`

## 3. 负载均衡模块

### 3.1 读节点选择（加权轮询入口有效）

- **Scenario**
  - 给出 3 个可选节点，执行一次读节点选择。
- **Use Case**
  - 验证负载均衡模块可从候选节点中选出可用节点，不出现空结果。
- **输入**
  - nodes：
    - `127.0.0.1:9012`
    - `127.0.0.1:9013`
    - `127.0.0.1:9014`
  - partition：`0`
- **预期输出**
  - `selectReadNode(...)` 返回非空 `NodeInfo`。

### 3.2 负载上报可被接受

- **Scenario**
  - 连续上报 3 个节点的负载数据。
- **Use Case**
  - 验证 `NodeMonitor` / `MigrationManager` 路径可接受并处理负载输入（不抛异常）。
- **输入**
  - `NodeLoad("127.0.0.1:9012", 30, 100, 50)`
  - `NodeLoad("127.0.0.1:9013", 35, 100, 40)`
  - `NodeLoad("127.0.0.1:9014", 20, 100, 20)`
- **预期输出**
  - `reportNodeLoad(...)` 调用过程不抛异常；
  - 测试整体通过。
- **对应实现**
  - 测试类：`minisql-common/src/test/java/com/zju/minisql/common/loadbalance/LoadBalancerImplTest.java`
  - 脚本：`scripts/test-a-loadbalance.ps1`

## 4. 本地存储引擎模块

### 4.1 插入后可按主键读取

- **Scenario**
  - 在临时目录中启动本地存储引擎，插入一条记录后点查。
- **Use Case**
  - 验证存储引擎基本 CRUD 中 `insert + get` 路径可用。
- **输入**
  - table：`student`
  - row：`primaryKey=1001`，列 `name=Alice`，partition=`0`
- **预期输出**
  - `get("student","1001")` 返回非空；
  - 读取到 `name == "Alice"`。

### 4.2 扫描可返回记录

- **Scenario**
  - 在已有一条数据的情况下执行默认扫描。
- **Use Case**
  - 验证扫描能力可拿到至少一条结果。
- **输入**
  - `ScanOptions` 默认值（无起止 key、无过滤、无限制）
- **预期输出**
  - `scan("student", opts)` 的迭代器 `hasNext() == true`。
- **对应实现**
  - 测试类：`minisql-worker/src/test/java/com/zju/minisql/worker/storage/LocalStorageEngineImplTest.java`
  - 脚本：`scripts/test-a-storage.ps1`

## 5. 脚本级执行与验收标准

四个脚本分别执行单模块定向测试：

- `powershell -ExecutionPolicy Bypass -File scripts/test-a-distribution.ps1`
- `powershell -ExecutionPolicy Bypass -File scripts/test-a-replica.ps1`
- `powershell -ExecutionPolicy Bypass -File scripts/test-a-loadbalance.ps1`
- `powershell -ExecutionPolicy Bypass -File scripts/test-a-storage.ps1`

补充的容错容灾回归脚本：

- `powershell -ExecutionPolicy Bypass -File scripts/test-replica-fix.ps1`
  - 覆盖副本管理 + 副本同步 RPC 两条修复链路；
- `powershell -ExecutionPolicy Bypass -File scripts/test-ha-dr-full.ps1`
  - 覆盖数据分布、副本管理（含自动切换重试）、副本 RPC、负载均衡、本地存储；
  - 可在本地直接运行，无需 Docker。

### 统一验收标准

- Maven 命令返回码为 `0`；
- 脚本输出包含对应“模块通过”提示；
- 无未处理异常中断。

## 6. 与主流程联调的关系

这些测试主要覆盖“模块级逻辑正确性”，而端到端联调覆盖“模块接线是否生效”：

- 模块测试通过 -> 证明各模块核心行为可用；
- 主流程联调通过 -> 证明模块已成功挂接到 `WorkerStarter` 与 `GroupBQueryDemo`，并可在真实进程中运行。
