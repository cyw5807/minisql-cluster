## 一、系统模块简介

本人在分布式MiniSQL系统中负责研发**本地存储引擎**、**数据分布**、**副本管理**和**负载均衡**四个模块，采用 Java 程序设计语言，在 ZooKeeper 分布式协调平台下进行架构设计、编码与调试。

**本地存储引擎模块**（LocalStorageEngine）是数据平面最底层模块。当前实现采用 **H2 MVStore** 作为分片存储，每个 partition 对应一个 `.db` 文件，行对象使用 Kryo 序列化后写入 `MVMap<String, byte[]>`，对上层提供统一的行级读写接口、谓词下推扫描接口及副本日志回放接口，具体功能如下：

1. 提供面向查询引擎的行级 CRUD 接口（`insert`/`delete`/`update`/`get`/`scan`）
2. 提供面向副本管理模块的日志回放接口（`applyReplicationLog`）
3. 提供面向负载均衡模块的分片导入导出接口（`exportPartition`/`importPartition`）
4. 管理本节点所有 Partition 的数据文件生命周期

**数据分布模块**（DistributionManager）负责读写路由与节点变更处理。当前实现中，分片号由 `hashCode(key) % 1024` 计算，优先从元数据读取 Primary，必要时回退到一致性哈希环选路，具体功能如下：

1. 通过 `partitionIdForKey` 计算分片号，并返回对应目标节点
2. 维护本地路由缓存（RoutingTable），减少 ZooKeeper 查询开销
3. 节点上下线时动态重建哈希环并失效缓存

**副本管理模块**（ReplicaManager）负责维护集群中数据的多副本写入、故障切换与节点恢复，具体功能如下：

1. 写入时先写 Primary，再顺序同步 Replica，并按 Quorum 判定成功；每条写入生成带单调递增 `logIndex` 的 `ReplicationLogEntry` 并缓存于内存日志缓冲区
2. 读取时在 Primary 和 Replica 之间进行轮询路由，分摊读压力
3. Primary 写失败时触发一次自动故障切换并重试写入
4. Replica 接收日志时检测 `logIndex` 连续性，发现缺口（gap）后向 Primary 请求补偿，补齐后再提交当前日志，最多循环重试 3 轮
5. 节点重连时主动按分区拉取缺失日志并全量追平（`onNodeRecovery`），对所有有日志的分区逐一发起 `recoverEntries`，失败自动重试（最多 5 次/分区，间隔 200ms），同时将恢复节点补回分区缓存

**负载均衡模块**（LoadBalancer）负责读请求分发与迁移触发，具体功能如下：

1. 读请求按加权轮询（Nginx平滑加权轮询算法）分发到各副本
2. 通过 `reportNodeLoad` 上报负载，超过阈值时触发迁移
3. 分片迁移按“复制（示意）→切换路由→清理（示意）”顺序编排

## 二、本地存储引擎模块实现说明

### 2.1 模块组件设计

本模块包含以下组件：

1. **LocalStorageEngine（接口）**：对外暴露统一的读写和迁移接口
2. **LocalStorageEngineImpl**：接口的具体实现，持有 PartitionManager 实例
3. **PartitionManager**：管理所有 Partition 的内存存储与二进制分片文件
4. **ScanOptions**：封装范围扫描参数，支持谓词下推
5. **Row**：行数据结构（`primaryKey/tableName/partitionId/columns`）
6. **ReplicationEntry**：副本日志条目，包含操作类型、数据及日志序号

**模块架构：**

```mermaid
graph TD
    QueryEngine[组员B\n查询引擎] -->|scan / get| LSE[LocalStorageEngine]
    ReplicaManager[副本管理模块] -->|applyReplicationLog| LSE
    DistributionManager[数据分布模块] -->|迁移时读取分片数据| LSE
    LSE --> PM[PartitionManager\n分区文件管理]
    PM --> MVS1[MVStore\npartition-0.db]
    PM --> MVS2[MVStore\npartition-1.db]
    PM --> MVSN[MVStore\npartition-N.db]
```

**技术选型：**

| 方案                                 | 优点                           | 缺点                         | 是否采用 |
| ------------------------------------ | ------------------------------ | ---------------------------- | -------- |
| MVStore（H2内置）                    | 纯 Java、持久化能力稳定、支持有序Map | 较 RocksDB 性能上限偏低         | **是**   |
| 内存Map + 文件序列化                  | 实现简单、调试方便               | 一致性与恢复能力弱，工程性不足 | 否       |
| RocksDB JNI                          | 性能高                          | JNI依赖复杂，部署麻烦         | 否       |

**关键参数：**

| 参数               | 当前值/策略                    | 说明                                      |
| ------------------ | ------------------------------ | ----------------------------------------- |
| 数据目录           | 构造时传入 `Path dataDir`       | 分片文件输出目录                          |
| 分片文件名         | `partition-{id}.db`             | 与 partitionId 一一对应                    |
| 行序列化方案       | Kryo                            | `LocalStorageEngineImpl` 对行对象进行序列化 |
| 分片存储结构       | MVStore + MVMap(`rows`)         | 每个分片一个 `MVStore`，数据写入 `MVMap`    |
| 扫描策略           | 全分片遍历 + key排序 + 谓词过滤  | 支持 `start/end/predicate/offset/limit`     |

## 三、数据分布模块实现说明

### 3.1 模块组件设计

本模块包含以下组件：

1. **DistributionManager（接口）**：对外暴露路由和集群变更通知接口
2. **DistributionManagerImpl**：一致性哈希路由的具体实现
3. **ConsistentHashRing**：一致性哈希环，使用 `TreeMap<Long, NodeInfo>` 存储虚拟节点
4. **RoutingTable**：本地路由缓存，TTL 30秒，集群变更时强制失效
5. **MetadataService（接口）**：封装 ZooKeeper 元数据读写操作（由组长实现）
6. **NodeInfo**：节点地址和状态描述

**模块架构：**

```mermaid
graph TD
    Client -->|routeForWrite / routeForRead| DM[DistributionManager]
    DM --> CHR[ConsistentHashRing]
    DM --> RT[RoutingTable Cache]
    DM --> MS[MetadataService\nZooKeeper Client]
    MS -->|读写分片元数据| ZK[(ZooKeeper)]
    ClusterManager -->|onClusterChange| DM
```

**一致性哈希原理：**

将哈希空间组织成环 $[0, 2^{32})$，节点和数据均映射到环上：

$$\text{partitionId} = \text{hash}(key) \mod 2^{32}$$

$$\text{node} = \text{首个顺时针方向上的虚拟节点}$$

每个物理节点映射为多个虚拟节点以解决数据倾斜：

$$\text{virtualNode}_i = \text{hash}(\text{nodeId} + \texttt{"/"} + i), \quad i \in [0, V)$$

推荐 $V = 150$。

**关键参数：**

| 参数             | 推荐值 | 说明                                     |
| ---------------- | ------ | ---------------------------------------- |
| 虚拟节点数 $V$   | 150    | 每个物理节点的虚拟节点数，越大分布越均匀 |
| RoutingTable TTL | 30s    | 缓存失效时间，节点变更时强制失效         |
| 分片总数         | 1024   | 固定分片数，与节点数解耦                 |

## 四、副本管理模块实现说明

### 4.1 模块组件设计

本模块包含以下组件：

1. **ReplicaManager（接口）**：对外暴露写入、读取及故障通知接口
2. **ReplicaManagerImpl**：持有三个子处理器的总协调类；维护 `partitionLogs`（内存日志缓冲）与 `partitionLogIndex`（递增序号），负责 `onNodeRecovery` 全量追平逻辑
3. **PrimaryHandler**：处理写入及到 Replica 的顺序同步与 Quorum 统计；内置 `syncWithRecovery` 循环负责被动 gap 修复；提供 `recoverReplica` / `getLastAppliedIndex` 供主动恢复路径调用
4. **ReplicaHandler**：委托 LoadBalancer 处理读取路由
5. **FailoverHandler**：处理 Primary 宕机时的 Replica 选举与元数据更新（内部维护同步进度Map）
6. **ReplicaSyncTransport（接口）**：副本同步传输抽象，扩展了 `recover` 和 `getLastAppliedIndex` 方法
7. **ReplicaSyncRpcTransport**：`ReplicaSyncTransport` 的 Netty RPC 实现，通过 `NettySyncClient` 调用 Worker 侧的 `ReplicaDataSyncService`
8. **ReplicationLogEntry**：日志条目，携带 `partitionId`、`tableName`、`row`、`logIndex`（单调递增）等字段
9. **ReplicaSyncAck**：同步响应，包含 `ok` / `gap` / `fail` 三种状态，`gap` 状态时携带 `expectedLogIndex` 供恢复路径定位缺口
10. **ReplicaDataSyncService（接口，Worker 侧）**：定义 Worker 暴露给 Primary 的三个 RPC 方法：`appendEntry` / `recoverEntries` / `getLastAppliedIndex`
11. **ReplicaDataSyncServiceImpl（Worker 侧实现）**：按 `logIndex` 连续性校验接收的日志条目，检测到 gap 返回 `ReplicaSyncAck.gap`；`recoverEntries` 按序去重回放，保证幂等

**副本模型：**

每个 Partition 采用 Primary-Backup 模型，推荐 ReplicationFactor = 3（1 Primary + 2 Replica）。

采用**顺序同步 + Quorum 判定**策略：

$$\text{写入成功条件：Primary写成功} + \text{ack} \ge \text{quorum}$$

其中在实现中：$\text{quorum} = 1 + \max(1, \lfloor \text{replicas}/2 \rfloor)$。

**副本状态机：**

```mermaid
stateDiagram-v2
    [*] --> FOLLOWER: 节点启动
    FOLLOWER --> SYNCING: 数据落后，开始同步
    SYNCING --> FOLLOWER: 同步完成
    FOLLOWER --> PRIMARY: 被选举为Primary
    PRIMARY --> FOLLOWER: 主动降级或检测到脑裂
    PRIMARY --> [*]: 节点宕机
    FOLLOWER --> [*]: 节点宕机
```

**模块架构：**

```mermaid
graph TD
    Client -->|write| RM[ReplicaManager]
    Client -->|read| RM
    RM --> PH[PrimaryHandler\n处理写入/被动gap修复]
    RM --> RH[ReplicaHandler\n处理读取路由]
    RM --> FH[FailoverHandler\n处理故障切换]
    PH -->|appendEntry RPC| Replica1[Worker Replica-2\nReplicaDataSyncServiceImpl]
    PH -->|appendEntry RPC| Replica2[Worker Replica-3\nReplicaDataSyncServiceImpl]
    PH -->|recoverEntries RPC| Replica1
    PH -->|getLastAppliedIndex RPC| Replica1
    FH --> MS[MetadataService\nZooKeeper]
    CM[WorkerDiscovery\n组长] -->|onNodeFailure| RM
```

**关键参数：**

| 参数                   | 推荐值        | 说明                                                          |
| ---------------------- | ------------- | ------------------------------------------------------------- |
| ReplicationFactor      | 3             | 每份数据存3份（1 Primary + 2 Replica）                        |
| Quorum                 | 2             | 至少2个节点确认写入才返回成功                                 |
| 同步策略               | 顺序调用      | 按 Replica 列表逐个调用 `syncWrite`                           |

| 节点恢复重试间隔       | 200ms         | 每次重试等待时间，应对 Worker 服务启动延迟                    |

## 五、负载均衡模块实现说明

### 5.1 模块组件设计

本模块包含以下组件：

1. **LoadBalancer（接口）**：对外暴露读节点选择和负载上报接口
2. **LoadBalancerImpl**：持有 WeightedRoundRobin、NodeMonitor 的协调类（内部创建 MigrationManager）
3. **WeightedRoundRobin**：实现 Nginx 平滑加权轮询算法，Primary 权重 1，Replica 权重 2
4. **NodeMonitor**：接收 `reportNodeLoad` 上报，超过阈值时触发迁移
5. **MigrationManager**：执行迁移编排（复制示意→切换路由→清理示意）
6. **NodeLoad**：节点负载信息载体

**模块架构：**

```mermaid
graph TD
    Client -->|读请求| LB[LoadBalancer]
    LB --> WRR[WeightedRoundRobin\n加权轮询]
    WRR -->|选中节点| Node[目标副本节点]

    Monitor[NodeMonitor\n负载上报触发检查] -->|超过阈值| MM[MigrationManager\n分片迁移]
    MM -->|复制数据| TargetNode[目标节点]
    MM -->|更新元数据| MS[MetadataService\nZooKeeper]
    MM -->|通知路由失效| DM[DistributionManager]
```

**加权轮询算法（Nginx平滑版）：**

$$W_{primary} = 1, \quad W_{replica} = 2$$

选节点公式（平滑加权轮询，避免同一节点连续被选中）：

$$\text{选中节点} = \arg\max_{i}(currentWeight_i), \quad currentWeight_i \mathrel{+}= weight_i$$

$$\text{选中后}: currentWeight_{selected} \mathrel{-}= \sum_{i} weight_i$$

**分片迁移触发条件：**

$$\text{触发迁移} \iff \frac{\text{节点数据量}}{\text{节点容量}} > \theta_{high} = 0.8$$

$$\text{迁移目标节点} = \arg\min_{i} \frac{\text{节点}i\text{数据量}}{\text{节点}i\text{容量}}$$

$$\text{迁移满足}: \frac{\text{迁移后源节点数据量}}{\text{容量}} < \theta_{low} = 0.6$$

**关键参数：**

| 参数                       | 推荐值 | 说明                          |
| -------------------------- | ------ | ----------------------------- |
| Primary读权重              | 1      | 低权重减少Primary读压力       |
| Replica读权重              | 2      | 高权重让Replica承担更多读请求 |
| 高水位阈值 $\theta_{high}$ | 0.8    | 节点数据量超过80%容量触发迁移 |
| 低水位阈值 $\theta_{low}$  | 0.6    | 迁移后源节点数据量目标低于60% |
| 监控方式                   | 被动上报 | 通过 `reportNodeLoad` 触发检查  |

## 七、开发体会

本次分布式MiniSQL系统的开发是我第一次在实际项目中综合运用分布式系统的核心概念，收获颇丰。

**关于一致性哈希的理解**：在设计数据分布模块时，我深刻体会到虚拟节点对解决数据倾斜问题的重要性。理论上均匀，实践中若虚拟节点数不足（如V=10），仍会出现明显的分布不均。将V调整到150后，实测各节点分片数量偏差降至10%以内。

**关于分布式一致性的权衡**：副本管理当前采用“Primary先写 + Replica顺序同步 + Quorum判定”的策略。这个实现路径清晰、便于调试，同时在Primary写失败时可自动触发一次故障切换与重试，在课程项目阶段很好地平衡了实现复杂度与可用性。

**关于模块间接口设计的重要性**：四个模块之间存在复杂的调用关系（如迁移完成后需要依次通知MetadataService和DistributionManager），清晰的接口契约大幅降低了联调成本。提前约定好接口文档，避免了大量的沟通返工。

**关于分层架构的价值**：LocalStorageEngine作为最底层模块，其接口稳定性直接影响到上层所有模块的开发进度。在存储层采用 MVStore 而非自研 B+ 树，有效降低了实现复杂度，使我们能够把主要精力投入到分布式路由、副本与迁移等核心逻辑。

**关于节点恢复机制的工程细节**：实现副本重连追平时遇到了两个典型问题。一是 ZooKeeper 元数据在故障切换后未同步更新旧 Primary 的副本归属，导致按元数据过滤分区时跳过了需要恢复的分区；解决方案是放弃元数据过滤，改为遍历本地内存日志缓冲（`partitionLogs`）的所有分区，确保不遗漏。二是恢复调用时 Worker 的 RPC 服务可能尚未就绪，导致第一次调用失败；引入 `recoverWithRetry`（5次重试，200ms间隔）后问题得以解决。这两处细节充分说明，在分布式系统中"故障假设"往往比"正常路径"更复杂，防御性设计和充分的重试机制是工程可靠性的基础。
